Account Takeover Through Manipulation of Session Storage

CLIENT-SIDE IDENTITY // USER ORACLE // FULL ATO

researcher
Naveen Jagadeesan
published
2023-03-30
platform
Financial Web Application

The app had a real session cookie. Akamai was in front. On paper, authentication looked fine. Identity for the SPA, though, lived in sessionStorage — a JSON blob the UI treated as gospel after login. If you can read another user’s object from an API and paste it into that key, you are not “hacking the cookie.” You are walking into their invoices as them.

This is that bug on a multi-tenant finance product: enumeration, client-side trust, OSINT for a real corporate admin, full account takeover.

Target

Web app for invoice generation, payment tracking, reconciliation. Users create orgs, invite teammates, assign roles. React frontend, REST APIs, cookie-based auth, Akamai WAF. After login you are pushed through org setup — normal B2B SaaS shape.

I registered a test account and watched the network tab during login more carefully than the “secure” branding deserved.

The API that talked too much

Early in the auth flow the client asked the backend whether a user existed. The answer was not a boolean. 

GET /api/v1/user?email=nj@tvhsecurity.com HTTP/2
Host: www.target.com
Cookie: session=blahblah
{
  "status": true,
  "message": "Successfully Retrieved",
  "result": [
    {
      "UserId": 601XXX,
      "EmailId": "nj@tvhsecurity.com",
      "RoleName": "Admin",
      "CountryName": "India",
      "Status": "Approved",
      "IsActive": true
    }
  ]
}

That is an enumeration oracle and a profile dump in one call: stable user id, role, country, approval state. Useful for the SPA. Equally useful for an attacker who only needs an email.

Where the SPA kept “who am I?”

After login, browser storage held a login (or equivalent) entry whose JSON matched the user API almost field-for-field.

Hypothesis: cookie proves “some authenticated browser session,” but the React app decides which user to render and which tenant APIs to call from session storage. If subsequent requests key off UserId / email from that blob without binding them to the server session, client-side swap equals impersonation.

I created a second legitimate account to test the theory cleanly before touching anyone real. 

GET /api/v1/user?email=nj+1@tvhsecurity.com HTTP/2
Host: www.target.com

Response again included the full object. I copied it into the first account’s session storage under the same key, refreshed, and the UI flipped identity. Same cookie. Different user. That is the whole vulnerability class in one demo.

Making it real: OSINT → employee admin

Demo accounts prove the bug. Reputation-grade reports need a realistic victim without crossing lines. I stayed in publicly available sources: contact pages, GitHub issues and comments, tracker noise — the places engineers accidentally leave work emails.

Candidate addresses went through the same user API (Intruder / content-length diffs work when responses vary for hit vs miss). One corporate mailbox resolved to an approved Admin: 

GET /api/v1/user?email=baXXXXXX@target.com HTTP/2
Host: www.target.com
{
  "status": true,
  "message": "Successfully Retrieved",
  "result": [
    {
      "UserId": 10XXX,
      "EmailId": "baXXXXXX@target.com",
      "RoleName": "Admin",
      "CountryName": "United States",
      "Status": "Approved"
    }
  ]
}

At that point the checklist was closed: valid email, full client-side identity object, proven sessionStorage trust, authenticated attacker session of my own. 

{
  "UserId": 10XXX,
  "EmailId": "baXXXXXX@target.com",
  "RoleName": "Admin",
  "CountryName": "United States",
  "Status": "Approved"
}

Paste → refresh → their workspace.

Visible impact in-session: invoices, financial records, user admin surfaces, the operational data you expect on a finance tenant. No password reset. No cookie forge. No XSS required for the final step — only an account of my own and a JSON edit.

Impact

Any user who can authenticate (or otherwise obtain a session that loads the SPA) and query /api/v1/user can:

  • Enumerate accounts and roles by email
  • Build a drop-in identity object for session storage
  • Impersonate arbitrary users in the UI and on any API that trusts client-supplied user identifiers from that context

Against finance workflows that means confidentiality of invoices and payment state, integrity risk on records, and lateral access inside orgs — especially when the victim is Admin.

Severity is high/critical depending on whether backend calls re-check session↔user binding. In this assessment, the practical result was full account takeover from the user’s perspective.

Root cause

  1. Client-side identity store — mutable sessionStorage treated as the source of truth for who is logged in.
  2. Session not bound to subject — auth cookie did not force all subsequent authorization decisions to a server-side user id.
  3. Oversharing user API — login-adjacent endpoint returned full role and status metadata to support the SPA, and doubled as an oracle.
  4. Enumeration — email in, rich object out, with observable differences for valid users.

Fix direction

  • Server session (or token sub) is the only identity. Client storage may cache display names; it must never be authoritative.
  • Every sensitive API derives userId from the session, never from a body field the SPA copied out of storage.
  • User lookup for login should return minimal data (exists / not exists is already risky; full role objects are worse). Prefer constant-time, uniform responses where product allows.
  • Drop verbose profile fields from pre-auth or broad-auth endpoints; load them only after server-side subject is fixed.

Notes from the assessment

I still open Application → Storage on every SPA before I trust the cookie story. WAFs do not inspect what React believes after setItem. Here the chain was almost boring once the hypothesis was right: talkative API, client-side user object, one refresh.

Boring bugs empty bank-adjacent tables. Report them like it.
