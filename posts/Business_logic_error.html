Price Manipulation After a Salted-Hash Fix

INCOMPLETE FIX // NEGATIVE QTY // ORDER MATH

researcher
Naveen Jagadeesan
published
2023-04-12
platform
E-Commerce Web Application

They already knew people had been tampering with checkout amounts. The fix was a saltedhash tied to the line so you could not just edit price and walk away. That is a reasonable patch for the bug they got burned on. It is not a model of the order.

I went back to the same class of flow after the hardening and asked a simpler question: what else still multiplies into amount_to_be_paid? Spoiler — quantity, and negative values were legal as far as the server was concerned.

Target

E-commerce style checkout: products/services, address, pay. React web client, mobile apps in the wild, AWS hosting. Bodies were encrypted on the wire; same story as a lot of regional commerce stacks — reverse the SPA crypto (key/IV/mode from JS), then work on clear JSON. Crypto is only the access method here; the finding is business logic.

What the hardened request looked like

After decrypt, checkout resembled: 

POST /api/v1/checkout HTTP/2
Host: www.target.com

{
  "saltedhash": "hashcode",
  "product_id": "121212",
  "price": "10",
  "quantity": "10",
  "address": "ABC city",
  "zipcode": "123456"
}
{
  "saltedhash": "hashcode",
  "product_id": "121212",
  "price": "10",
  "quantity": "10",
  "amount_to_be_paid": "100",
  "address": "ABC city",
  "zipcode": "123456"
}

amount_to_be_paid is clearly derived. The hash is there to stop the client from inventing a price the catalog never offered. Direct price edits failed integrity checks — good.

Attack the formula, not the constant

If payable ≈ f(price, quantity, …) and only price is integrity- protected, quantity is still a lever. I left price and saltedhash alone and sent a negative quantity: 

POST /api/v1/checkout HTTP/2
Host: www.target.com

{
  "saltedhash": "hashcode",
  "product_id": "121212",
  "price": "10",
  "quantity": "-5",
  "address": "ABC city",
  "zipcode": "123456"
}
{
  "saltedhash": "hashcode",
  "product_id": "121212",
  "price": "10",
  "quantity": "-5",
  "amount_to_be_paid": "6",
  "address": "ABC city",
  "zipcode": "123456"
}

Hash still accepted. Price untouched. Payable collapsed (here to 6 in the lab case — absolute numbers depend on other fees/taxes in the basket). The server recalculated using the client-supplied quantity instead of rejecting non-positive units.

They integrity-protected a field. They did not integrity-protect the order.

Why this keeps happening

Incident response often patches the parameter named in the last report. Scanners and junior test cases poke price, amount, total. Quantity, weight, tax flags, coupon stacking, currency codes — same math, less attention. Negative or zero quantity is an old trick; it still works when the only new control is “HMAC the unit price.”

Impact

  • Arbitrary deflation of amount due at checkout
  • Purchase of goods/services below catalog economics
  • Downstream mess for inventory, fraud, and reconciliation if orders still fulfill

Business impact scales with whether payment capture uses the tampered total and whether fulfillment is automatic. In testing, the calculation stage already accepted the abusive total — that is enough for a solid critical/high business-logic report on most programs.

Root cause

  1. Integrity control scoped to price (or price-derived hash inputs that omitted quantity).
  2. Missing domain rules: quantity ∉ ℤ⁺ should never reach the pricing function.
  3. Server trusted client numerics for any factor of the total not covered by the hash.
  4. No final invariant: “recompute total from catalog price × server-side qty and compare to charged amount.”

Remediation

  • Reprice entirely server-side from product_id + allowed quantity; client price is display-only.
  • If you keep a hash/MAC, include every client-influenced input that affects money — or better, stop accepting those inputs.
  • Validate quantity as a positive integer within stock limits before any money math.
  • Reject the whole checkout on constraint failure; do not clamp silently unless product explicitly wants that.

Closing

When a program says “we fixed payment tampering,” retest the whole formula. The salted hash did its job on price. The order still spoke JSON, and quantity: -5 was enough to argue with the cash register.
