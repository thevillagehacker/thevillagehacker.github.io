Hotel Subscription Exploitation through Business Logic Errors

PAYMENT LOGIC BYPASS // ENCRYPTED API ANALYSIS // MEMBERSHIP ABUSE

researcher
Naveen Jagadeesan

2023-10-31
platform
Hotel Booking Platform

The product team had wrapped every REST body in AES. Cloudflare sat in front. On a B2B hotel booking platform that looks “secure” in a recon report. It is not. Client-side encryption with a key shipped in the JS bundle is transport cosplay — once you can decrypt, you are editing business parameters the server still trusts.

This writeup is the money bug on that stack: free Diamond membership, 12% off bookings, loyalty multipliers, no extra charge. The crypto reverse is the door; the broken payment state machine is the room. For the pure RE notes on the same family of apps, see the client-side encryption analysis.

Target

B2B hotel reservation product. Bookings plus a paid club with four tiers — Silver, Gold, Platinum, Diamond. Diamond was the interesting one in the UI: roughly 12% off bookings, 3× loyalty points, short promotional membership windows attached to checkout flows.

Stack from the outside: React SPA, REST APIs on separate hosts, AWS-side hosting, Cloudflare WAF. None of that blocked reading the browser’s own secrets.

Why the traffic looked opaque

Request and response bodies were ciphertext. Same shared key material on client and server — classic “encrypt the JSON so Burp users give up” pattern. You cannot meaningfully test checkout logic until you can round-trip encrypt yourself.

The org also reused a CDN config blob across multiple apps, so one bundle analysis mapped more than one property. That is an architecture smell as much as a finding.

Wire format was effectively: Base64(AES-CBC(JSON)) + "---" + Base64(IV). Static IV is its own problem; for this assessment the practical win was offline decrypt → edit → re-encrypt with the same parameters the SPA used.

Membership on the checkout path

Club purchase was bolted onto booking. New-ish feature surface — multiple payment workflows, internal codes, membership flags traveling with the order. After decrypt, one field stood out: payment_method. It did not look like a card brand. It looked like a workflow selector.

Adding membership hit endpoints in this shape (bodies encrypted on the wire): 

PUT /api/v18/add_membership HTTP/2
Host: api.target.com

{
  "data": "EncryptedPayload---Base64IV"
}
POST /api/v16/purchase_add_ons HTTP/2
Host: api.target.com

{
  "data": "EncryptedPayload---Base64IV"
}

Intended happy path for the promo I was testing: pay for the room and the Diamond add-on → two months of Diamond, 12% discount semantics, loyalty boost.

The logic bug

I compared payloads across payment paths and flipped payment_method from 26 to 21 inside the decrypted JSON, then re-encrypted. The membership-related flags stayed true. The extra line item for the club did not.

After the add-on purchase call, activation parameters still looked like a successful club grant: 

{
  "user_club_membership_id": 123456,
  "diamond_club": true
}

Backend behavior matched the flags, not a server-side ledger check that “Diamond surcharge was captured.” Booking completed. Diamond benefits applied. No additional membership payment.

Encryption only raised the cost of reading the parameters. It did not make those parameters trustworthy.

Impact

For an attacker who can reverse the SPA crypto (anyone with DevTools):

  • Activate paid club tiers without paying the membership fee
  • Apply Diamond-level booking discounts (observed ~12%)
  • Accrue inflated loyalty multipliers
  • Repeat across bookings while the broken path remains open

This is pure revenue and integrity loss — not RCE, but it is the kind of bug finance and product teams care about more than another reflected XSS.

Root cause

  1. Client-held secrets — AES key/IV and algorithm in distributed JS; “encrypted API” was reversible by design.
  2. Workflow IDs as client input — payment_method changed server payment behavior without a matching authorization from a payment provider event.
  3. Entitlement decoupled from settlement — diamond_club: true (and membership IDs) were honored without proving the membership SKU was paid.
  4. Feature glue — subscription bolted onto booking checkout without an end-to-end invariant: “club benefits require successful club payment in this transaction.”

Remediation

  • Derive membership state only from server-side payment records / provider webhooks, never from a client flag in an encrypted blob.
  • Treat payment_method and similar enums as hints at most; map them through allowlisted server workflows with server-priced line items.
  • Stop shipping long-lived symmetric API keys in the frontend. If you need field-level protection, use per-session keys from a hardened channel — and still never trust decrypted client JSON for money.
  • Add integration tests: every path that sets diamond_club must assert a captured charge for that SKU.

Takeaway

I see this pattern a lot on travel and marketplace stacks: new loyalty or subscription SKU, old checkout, “we’ll encrypt the body so partners feel safe.” The interesting bugs are not in AES-CBC. They are in the two-digit workflow codes and boolean entitlements that survive the round-trip.

Decrypt the boring layer. Then attack the state machine.
