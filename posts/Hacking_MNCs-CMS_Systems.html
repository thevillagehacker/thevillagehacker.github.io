CMS Privilege Escalation via Cognito and localStorage Roles

HIDDEN SIGNUP // EMPTY GROUPS // LOCALSTORAGE RBAC

researcher
Naveen Jagadeesan
published
2023-03-25
platform
ReactJS / AWS Cognito

Corporate CMS, Cognito login only, no “Sign up” in the UI — the kind of app where content ships straight to public brand sites. Access was supposed to be invite-only. I still got a self-registered account, an empty cognito:groups claim, and publisher powers by typing a string into localStorage.

Two misconfigurations, one chain: forgotten registration surface, then authorization implemented in the browser.

Target

CMS used by a large org to manage product/service content across business-unit websites. React SPA, Amazon CDN, REST APIs, AWS Cognito for auth. Publish actions were not academic — approved content went live on production properties.

Marketing page: login form. No registration. That is a claim, not a control.

Misconfig 1 — signup without a door

JS bundles did not hand me a friendly route map, so I fuzzed paths. Hit: 

https://abc.com/abc/xyz/signup.html

Legacy or staging-era registration, still live. Account creation worked without email verification, admin approval, or any invite token I could see. Cognito (or the app’s wrapper) happily authenticated the new user.

CMS chrome stayed locked. Expected for a user with no groups — annoying if you stop there, useful if you treat it as “I now have a valid principal.”

What the JWT actually said

Login returned a JWT. Decoded payload was honest about the failure mode: 

{
  "sub": "e03f561e-9392-437d-abf7",
  "cognito:groups": [
    ""
  ],
  "cognito:username": "test",
  "cognito:roles": [
    ""
  ],
  "email": "nj@tvhsecurity.com"
}

Authenticated, zero roles. I tried the usual JWT games — injecting group names, alg confusion where applicable. Modified tokens died on signature/validation. Cognito was doing its job on the token. Something else was doing authorization.

Learning the role vocabulary

With a valid token I pulled more frontend assets and scraped embedded URLs (curl, httpx, a small URL extractor). Response- length outliers turned up a static page that documented internal CMS roles. Two that mattered:

  • editor — create / modify content
  • publisher — approve and push to production paths

So I had: principal, token, and the exact strings the product used for privilege. Still no server-side group membership.

Misconfig 2 — roles in localStorage

DevTools → Application → Local Storage. Key: user_role. Empty for my self-registered user. 

localStorage.setItem("user_role", "editor")

Refresh. Editor UI unlocked. APIs that the editor screens called accepted the session. No new Cognito group appeared in the JWT — the SPA (and, critically, backend handlers that believed the client) had moved on without it. 

localStorage.setItem("user_role", "publisher")

Refresh again. Approve / publish / reject flows available. That is production content control from an account nobody invited.

Cognito answered “who logged in?” The browser answered “what may they do?” Only one of those should matter for authorization.

Impact

  • Unauthenticated (or rather, uninvited) account creation on an internal CMS
  • Privilege escalation from no-role user → editor → publisher
  • Ability to alter and publish content destined for public properties
  • Trust/brand damage scenarios, defacement, malicious content injection on official sites

For an MNC CMS that is not a theoretical “IDOR on profile.” It is a path to the homepage of business units.

Root cause

  1. Registration endpoint left enabled without product exposure or invite gates.
  2. Authorization decisions driven by localStorage (and likely request fields derived from it) instead of Cognito groups / server-side RBAC.
  3. APIs that only checked “valid JWT” rather than “JWT group allows this action.”
  4. Role names discoverable from static assets — not a bug alone, but it removes guesswork once client trust exists.

Remediation

  • Disable or protect signup; require admin invite + verified email for CMS tenants.
  • Authorize every mutating CMS API from server-side claims (Cognito groups, DB roles keyed by sub). Ignore client-supplied role headers/body/storage.
  • Treat SPA role keys as UI hints only; refetch permissions from a server endpoint that cannot be spoofed by setItem.
  • Regression test: user with empty groups must receive 403 on editor/publisher routes even if user_role is forged.

Closing

I did not break Cognito. I walked around it. Hidden signup gave me a badge; localStorage printed the clearance level. If your CMS “uses Cognito,” open the storage panel anyway — a lot of enterprise React apps still keep a shadow RBAC in the browser and forget which one the API believes.
